# 專案後續實施 Roadmap

## 目標
- 穩定後台設定與金流流程，避免設定值不一致。
- 提升可維護性（降低重複邏輯、提高測試覆蓋）。
- 建立可持續交付流程（觀測、回滾、資料治理）。

## Phase 1（1-2 週）穩定性與防回歸
1. 設定系統防回歸
- 為 `updateSettings/getSettings` 增加整合測試（特別是 `linepay_sandbox`、`delivery_options_config`）。
- 建立設定鍵白名單文件，避免前後端鍵名漂移。
- 驗收：設定欄位可 round-trip（儲存後重新讀取一致）。

2. 前端設定頁健壯化
- 將布林值解析抽成共用 helper（main/dashboard 共用）。
- 針對「缺值/舊值格式」加入降級策略與提示。
- 驗收：重新整理、換分頁、慢網路下狀態一致。

3. CI 防線補強
- 在 CI 加入 `settings smoke`（呼叫 API 檢查關鍵設定鍵）。
- 驗收：關鍵設定鍵缺失時 CI 直接失敗。

## Phase 2（2-4 週）金流可靠度
1. LINE Pay 交易完整性
- 增加 request/confirm/cancel 的追蹤欄位（requestId、transactionId、attempt）。
- 補齊支付狀態機（pending -> authorized -> paid -> refunded）。
- 驗收：每筆訂單可追溯完整支付生命週期。

2. 異常處理與重試策略
- 針對超時/網路錯誤建立可觀測重試機制（含重試次數上限）。
- 新增後台「支付異常工單」清單。
- 驗收：異常訂單可在後台一鍵重送/標記結案。

## Phase 3（4-6 週）資料治理與可觀測性
1. 設定變更審計
- 新增 `settings_audit_logs`（操作者、變更前後、時間、來源 IP）。
- 驗收：後台可查每次設定變更紀錄。

2. 監控告警
- API 錯誤率、支付失敗率、Webhook 延遲告警。
- 驗收：異常在 5 分鐘內通知。

3. 安全與密鑰治理
- 制定密鑰輪替 SOP（LINEPAY、JWT、SMTP）。
- 驗收：可在不停機情況下完成密鑰輪替。

## Phase 4（6-10 週）商務功能擴展
1. 促銷引擎升級
- 支援「門檻 + 指定規格 + 時段」複合規則。
- 驗收：後台可視化設定複合促銷並可預覽試算。

2. 訂單營運分析
- 建立日/週報（轉換率、客單價、支付分佈、配送分佈）。
- 驗收：後台可下載 CSV 與查看趨勢圖。

3. 顧客體驗
- 會員預設資料優化、再購快捷下單、常用配送方式記憶。
- 驗收：重複購買流程點擊數下降。

## 優先執行建議
1. 先做 Phase 1（立即降低回歸風險）。
2. 接續 Phase 2（金流可靠度直接影響營收）。
3. 再做 Phase 3/4（提升長期效率與成長）。
